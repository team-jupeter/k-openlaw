Private Data Vault — 3계층 구조

각 개인 및 기관이 보유하는 암호화된 개인 데이터 저장소(PDV, 부호 90)는 법적 분쟁과 관련될 수 있는 모든 일상·업무 데이터를 24시간 365일 수집·저장하고, 저장 즉시 OpenHash 네트워크에 등록합니다.

설계 원칙 — PDV는 "사후 증거 수집"이 아닌 "사전 무결성 확립" 방식으로 작동합니다. 분쟁이 발생하기 훨씬 전부터 모든 데이터의 해시가 분산 원장에 등록되어 있으므로, 위변조는 기술적으로 불가능합니다. OpenLaw에서 증거 검증이 수초 내에 완료되는 이유입니다.

① 데이터 입수 계층

사용자 단말(부호 50)에서 발생하는 모든 법적으로 유의미한 데이터를 실시간 수집합니다.

  • 계약서 · 거래명세서
  • 이메일 · 메신저 대화 기록
  • 행정기관 발급 문서
  • 영수증 · 금융 거래 기록
  • 스마트폰 · PC 등 모든 단말 지원
② 해시 생성 및 등록 계층

수집된 각 데이터에 SHA-256 해시값과 타임스탬프를 연결하여 OpenHash 네트워크의 확률적 계층 선택 알고리즘(모듈 300)으로 전송합니다.

  • SHA-256 이중 재해싱 처리
  • 타임스탬프 ±5분 범위 검증
  • 중요 법적 문서 → Layer 3~4 우선 등록
  • BLS 서명으로 등록 사실 증명
③ 접근 통제 계층

PDV에 대한 유일한 접근 주체는 해당 사용자의 AI 비서(PAA/IAA)입니다. 법원 LLM을 포함한 제3자는 원본 데이터에 직접 접근할 수 없습니다.

  • PAA(92) — 개인 전용 Gate-Keeper
  • IAA(93) — 기관 전용, 개인 PDV 접근 불가
  • 사용자 명시적 동의 없이 데이터 전달 불가
  • AES-256 암호화 · JWT + 2FA 인증

※ 4계층 구조 기준 확률 분포. SHA-256의 암호학적 무작위성으로 공격자가 특정 계층을 예측할 확률: 2⁻²⁵⁶

확률적 계층 선택 알고리즘 (모듈 300) — 7단계

문서 해시값(400)과 타임스탬프를 SHA-256으로 이중 재해싱하여 어느 계층에도 예측 불가능하게 등록합니다. 이것이 위변조를 원천 차단하는 수학적 기반입니다.

SHA-256 이중 재해싱 절차
H₁ = SHA-256(문서_내용)← 1차 해싱 (문서 해시값 400)
H₂ = SHA-256(H₁ ‖ 타임스탬프)← 1차 재해싱
H₃ = SHA-256(H₂)← 2차 재해싱
계층 = LayerTable[H₃ mod 100]← 범위 변환 후 확률 테이블 매핑
공격자 특정 계층 예측 확률: 2⁻²⁵⁶ (사실상 0)

계층 간 상호 검증 — 하향식 + 상향식 이중 감시

하향식 검증 모듈 (310) — 상위→하위 감시
  • BLS 서명 검증
    e(서명, g) = e(H(메시지), 공개키) — 쌍선형 사상 등식 확인
  • Merkle Proof 검증
    무작위 트랜잭션에 Merkle Path(440) 요청 후 일치 확인
  • 타임스탬프 검증
    현재 시각 ±5분 범위 외 자동 거부 — 재전송 공격 방어
  • 검증 실패(610)
    해당 노드 즉시 격리(620), 15초 내 자동 복구(630)
상향식 감시 모듈 (320) — 하위→상위 감시
  • 응답 지연 감지
    평균 대비 3배 이상 지연 시 이상 신호
  • 서명 검증 실패율
    5% 초과 시 상위 계층에 자동 경보
  • 데이터 불일치 감지
    동일 시점 상이한 데이터 전송 탐지
  • 이상 감지 시
    즉시 연결 차단(620), 24시간 블랙리스트 등재

LPBFT 합의 메커니즘 (모듈 340) — Byzantine Fault Tolerance

자원 제약 환경을 위한 경량화 합의 알고리즘으로, n개 노드 중 f개의 악의적 노드가 있더라도 n ≥ 3f + 1 조건에서 합의를 보장합니다.

Byzantine Fault Tolerance 조건 및 합의 단계
Byzantine 내성 조건: n ≥ 3f + 1(n = 전체 노드, f = 악의적 노드)
합의 필요 응답 수: 2f + 1

Pre-prepare (220→230) → Prepare (2f+1 응답) → Commit (최종 확정)
패킷: <sequence_number, transaction_digest, signature> · ECDSA P-256 · 대역폭 50% 절감
증거 인출 프로토콜

PAA / IAA 증거 인출 6단계

원고 또는 피고의 AI 비서가 PDV에서 증거를 추출하여 법원 LLM(15)에 전달하는 표준 절차입니다. 모든 단계는 사용자 동의 기반으로 이루어집니다.

1

사용자 명령 수신 PAA / IAA

원고 또는 피고가 AI 비서(PAA, 92)에게 분쟁 상황을 설명하고 소송 제기 또는 방어 준비를 명령합니다. 자연어 입력을 NLP로 분석하여 법적 분쟁 맥락을 파악합니다.

2

PDV 내 증거 자동 탐색 NLP

분쟁 내용을 NLP로 분석하여 PDV(90) 내에서 관련 증거를 자동 탐색합니다. 날짜 범위, 당사자, 거래 유형 등 복합 조건으로 필터링합니다.

3

사용자 동의 취득 GDPR 준수

추출된 증거 목록을 사용자에게 제시하고, 각 항목의 제출 여부에 대해 명시적 동의를 취득합니다. 동의하지 않은 데이터는 어떠한 경우에도 외부로 전달되지 않습니다.

4

OpenHash 무결성 검증 BLS · Merkle

동의된 각 증거 항목에 대해 OpenHash 네트워크(91)에 등록된 해시값을 조회하고, BLS 서명 및 Merkle Proof로 일치 여부를 검증합니다. 불일치 항목은 자동 배제됩니다.

5

증거 패키지 JSON (94) 생성 정형화

검증 완료된 증거 항목들을 정형화된 JSON(94)으로 생성합니다. SHA-256 해시, 등록 계층, BLS 서명, 타임스탬프, 검증 결과가 모두 포함됩니다.

6

법원 LLM (15) 전달 및 재확인 최종

증거 패키지 JSON을 법원 LLM(15)에 전달합니다. 법원 LLM은 OpenHash 해시 일치를 재확인하고, 위변조 의심 증거를 자동 배제합니다. 최종 판정문 JSON에는 증거 패키지의 OpenHash 해시값이 링크되어 분석 근거 무결성이 내장됩니다.

법적 의의

PDV + OpenHash의 법적 함의

OpenHash는 "디지털 공증인"으로 기능합니다. 등록 시점의 사실 여부(진실성)는 데이터 생산자의 신뢰도에 의존하지만, 등록 시점 이후의 무변조(무결성)는 기술적으로 100% 보장합니다.

①

사실 확인 절차 간소화

OpenHash 검증 완료 증거에 대해서는 법정에서 진위를 다툴 수 없습니다. 등록 시점 이후의 무변조가 수학적으로 보장되어 사실 확인 단계가 수초 내에 완료됩니다.

②

위변조 의심 증거 자동 배제

OpenHash 등록값과 현재 해시값이 불일치하는 증거는 논증 과정에서 자동 배제됩니다(증거 패키지 JSON의 "배제_항목" 필드). 증거 조작 시도 자체가 무력화됩니다.

③

판결문 증거 무결성 내장

최종 판정문 JSON에 증거 패키지 JSON(94)의 OpenHash 해시값이 링크됩니다. 분석 근거 증거의 무결성이 판결문 자체에 내장되어 사후 번복 가능성을 기술적으로 차단합니다.
